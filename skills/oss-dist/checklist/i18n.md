# i18n & Multilingual Strategy

## When to Use

- User asks "how do I add translations to my README?"
- User asks "which languages should I target first?"
- User asks "how do I internationalize my CLI/library/app?"
- User wants to set up a community translation workflow
- User asks about i18n tooling (Crowdin, Weblate, GNU gettext, Fluent)

## Part 1: Documentation Multilingual

### Structure Selection by Scale

| Languages | Structure | Why |
|-----------|-----------|-----|
| 2–3 | Root `README.<lang>.md` + `docs/<lang>/` | Minimal noise, GitHub auto-renders README |
| 5+ | `docs/i18n/<lang>/` | Isolates translations, CI scoping, scales cleanly |

Rules that apply regardless of structure:
- English is the canonical source of truth — always authoritative, always updated first
- Every translation file must declare its source commit at the top
- Never let translations get ahead of English
- Use ISO 639-1 codes: `ko`, `ja`, `zh-Hans`, `zh-Hant`, `es`, `fr`, `de`, `pt`, `ru`, `it`

### Structure A: Small Scale (2–3 languages)

```
/
├── README.md                  # English (source of truth)
├── README.ko.md
├── README.ja.md
├── CONTRIBUTING.md
├── CONTRIBUTING.ko.md
└── docs/
    ├── architecture.md
    └── ko/
        └── architecture.md
```

Language switcher — place at the very top of every README variant:

```markdown
**[English](README.md)** | [Korean](README.ko.md) | [Japanese](README.ja.md)
```

### Structure B: Large Scale (5+ languages)

```
/
├── README.md                  # English only (with language switcher)
├── CONTRIBUTING.md
├── CHANGELOG.md
└── docs/
    └── i18n/
        ├── ko/
        │   ├── README.md
        │   ├── CONTRIBUTING.md
        │   └── architecture.md
        ├── ja/
        │   └── ...
        ├── zh-Hans/
        ├── zh-Hant/
        ├── es/
        ├── fr/
        ├── de/
        ├── pt/
        ├── ru/
        └── it/
```

Language switcher — link to `docs/i18n/` paths:

```markdown
[English](../../README.md) | [Korean](./ko/README.md) | [Japanese](./ja/README.md) | [Simplified Chinese](./zh-Hans/README.md) | [Traditional Chinese](./zh-Hant/README.md) | [Spanish](./es/README.md) | [French](./fr/README.md) | [German](./de/README.md) | [Portuguese](./pt/README.md) | [Russian](./ru/README.md) | [Italian](./it/README.md)
```

Every translated README must include at the top:

```markdown
> This is a translation of [README.md](../../README.md).
> The English version is the authoritative source and may be more up-to-date.
```

### Translation Automation

#### Option 1: LLM-based Auto-Translation (recommended)

Automatically generate translation PRs when English docs change. Uses any LLM API (Claude, GPT, etc.).

```yaml
# .github/workflows/i18n-translate.yml
name: Auto-translate docs
on:
  push:
    branches: [main]
    paths: ['README.md', 'CONTRIBUTING.md', 'docs/*.md']

jobs:
  translate:
    runs-on: ubuntu-latest
    permissions:
      contents: write
      pull-requests: write
    steps:
      - uses: actions/checkout@v4

      - name: Translate to all languages
        env:
          LLM_API_KEY: ${{ secrets.LLM_API_KEY }}
        run: |
          for lang in ko ja zh-Hans zh-Hant es fr de pt ru it; do
            mkdir -p docs/i18n/$lang
            # Call LLM API to translate each file
            # Preserve markdown formatting, code blocks, and frontmatter
          done

      - name: Create translation PR
        uses: peter-evans/create-pull-request@v5
        with:
          title: 'docs(i18n): update translations'
          branch: docs/i18n-update
```

Translation prompt template for LLM:

```text
Translate the following markdown document to {language}.
Rules:
- Preserve all markdown formatting exactly
- Do NOT translate code blocks — keep them as-is
- Do NOT translate URLs or image paths
- Keep technical terms in English where the target language convention prefers it
- Preserve YAML frontmatter structure, only translate title and description values
- Maintain the same file structure and heading hierarchy
```

#### Option 2: Crowdin (community-managed, enterprise)

- Free for OSS projects (crowdin.com/open-source)
- GitHub integration: auto-pull English strings, auto-PR translations
- Supports JSON, YAML, XLIFF, GNU gettext, Android XML, iOS .strings

```yaml
# crowdin.yml
project_id: "your-project-id"
api_token_env: CROWDIN_TOKEN
base_path: "."
files:
  - source: /README.md
    translation: /docs/i18n/%locale%/README.md
```

Also available as a GitHub Action:

```yaml
- uses: crowdin/translate-readme@v0.1.2
  with:
    destination: ./docs/i18n/
    language_switcher: true
```

#### Option 3: Community Contributions (manual)

Translation PR template (`PULL_REQUEST_TEMPLATE/translation.md`):

```markdown
## Translation PR

- **Language:** [ko / ja / zh-Hans / ...]
- **Source commit:** [SHA of English doc this translates]
- **Checklist:**
  - [ ] Translated from latest English version (commit above)
  - [ ] Technical terms kept in English where appropriate
  - [ ] Code blocks unchanged
  - [ ] Links updated if locale-specific resources exist
```

GitHub labels for translation management:

- `translation` — any translation work
- `help wanted` — open for community contribution
- `i18n/ko`, `i18n/ja`, etc. — language-specific

### Staying in Sync

Track translation staleness with a comment at the top of each translation file:

```markdown
<!-- Translated from README.md @ commit abc1234 (2025-03-01) -->
<!-- If English README has changed since then, this translation may be outdated -->
```

For Structure B, CI can scope checks to `docs/i18n/` only:

```yaml
on:
  pull_request:
    paths: ['docs/i18n/**']
```

### CI/CD Validation for Translation PRs

Automatically verify translation PRs:

- Frontmatter preserved (only `title`, `description` translated)
- File structure matches source (same filenames, same directories)
- Code blocks unchanged (diff against source)
- No empty or suspiciously short files
- Internal links still valid

## Part 2: Software i18n/l10n

### Terminology

| Term | Definition |
|------|------------|
| i18n | Internationalization — designing software to support multiple locales |
| l10n | Localization — adapting content for a specific locale |
| Locale | Language + region code: `ko-KR`, `zh-CN`, `pt-BR` |
| Translation key | Identifier for a string: `error.not_found`, `button.submit` |

### Format Selection by Stack

| Stack | Format | Tool |
|-------|--------|------|
| Web / React | JSON (react-i18next) | i18next |
| Web / Vue | JSON (vue-i18n) | vue-i18n |
| Web / Svelte | JSON | sveltekit-i18n |
| Rust CLI | TOML / Fluent | fluent-rs |
| Go CLI | JSON / GNU gettext (.po) | go-i18n |
| Python | GNU gettext (.po/.pot) | Babel |
| Node CLI | JSON | i18next |
| iOS/macOS | .strings / .xcstrings | Xcode built-in |
| Android | XML (res/values-{locale}/) | Android Studio |

### Key Design Rules

**Extraction — never hardcode strings:**

```rust
// ❌
println!("File not found: {}", path);

// ✅
println!("{}", t!("error.file_not_found", path = path));
```

**Pluralization — never string-concatenate:**

```javascript
// ❌
`${count} item${count !== 1 ? 's' : ''} found`

// ✅ (i18next)
t('items_found', { count })  // locale handles plural forms
```

**Date / number formatting — always use locale-aware APIs:**

```javascript
// ❌
new Date().toLocaleDateString()  // uses system locale

// ✅
new Intl.DateTimeFormat(userLocale, { dateStyle: 'long' }).format(date)
new Intl.NumberFormat(userLocale).format(1234567)
```

**RTL support (Arabic, Hebrew, Persian):**
- Use logical CSS properties: `margin-inline-start` not `margin-left`
- Add `dir="rtl"` to `<html>` or container
- Test with browser RTL mode or `direction: rtl` in DevTools

### Locale File Structure

```
locales/
├── en.json          # source of truth
├── ko.json
├── zh-CN.json
├── ja.json
└── es.json
```

Namespace for large projects:

```
locales/
├── en/
│   ├── common.json
│   ├── errors.json
│   └── onboarding.json
└── ko/
    ├── common.json
    └── errors.json
```

## Part 3: Translation Tooling

### Crowdin (recommended for community-managed)

- Free for OSS projects (crowdin.com/open-source)
- GitHub integration: auto-pull English strings, auto-PR translations
- Supports JSON, YAML, XLIFF, GNU gettext, Android XML, iOS .strings

```yaml
# crowdin.yml
project_id: "your-project-id"
api_token_env: CROWDIN_TOKEN
base_path: "."
base_url: "https://api.crowdin.com"
files:
  - source: /locales/en.json
    translation: /locales/%locale%.json
```

### Weblate (self-hosted alternative)

- Open-source, self-hostable
- Better for internal teams or privacy-sensitive projects
- Supports all major formats

### GitHub-only Workflow (minimal)

For small projects with few strings:

1. Create `locales/en.json` as source
2. Label translation PRs with `translation`
3. Use the PR template above
4. Require native speaker review for each locale

## Part 4: CLI-Specific i18n

### Go (go-i18n)

```go
bundle := i18n.NewBundle(language.English)
bundle.RegisterUnmarshalFunc("json", json.Unmarshal)
bundle.LoadMessageFile("locales/en.json")
bundle.LoadMessageFile("locales/ko.json")

localizer := i18n.NewLocalizer(bundle, "ko", "en")
msg := localizer.MustLocalize(&i18n.LocalizeConfig{
    MessageID: "error.not_found",
})
```

### Rust (rust-i18n)

```toml
# Cargo.toml
[dependencies]
rust-i18n = "3"
```

```rust
rust_i18n::i18n!("locales", fallback = "en");

fn main() {
    rust_i18n::set_locale("ko");
    println!("{}", t!("greeting.hello"));
}
```

### Node.js CLI (i18next)

```javascript
import i18next from 'i18next';
import Backend from 'i18next-fs-backend';

await i18next.use(Backend).init({
  lng: process.env.LANG?.split('.')[0] ?? 'en',
  fallbackLng: 'en',
  backend: { loadPath: './locales/{{lng}}.json' },
});

console.log(i18next.t('error.not_found'));
```

### Locale Detection (CLI)

```bash
# User's locale from environment (Unix)
echo $LANG          # en_US.UTF-8
echo $LC_ALL        # ko_KR.UTF-8

# Priority: $LC_ALL > $LANG > $LANGUAGE > fallback "en"
```

## Part 5: Checklist

### Documentation Multilingual
- [ ] Structure chosen by scale: root `README.<lang>.md` (2–3 langs) or `docs/i18n/<lang>/` (5+ langs)
- [ ] Language switcher at top of every translated file
- [ ] Every translation declares authoritative source link
- [ ] Source commit tracked in translation file header
- [ ] English updated first — translations never ahead of source
- [ ] `translation` + `help wanted` labels created
- [ ] PR template for translations added
- [ ] CI notification when English docs change
- [ ] LLM auto-translation workflow configured (optional, recommended for 5+ langs)
- [ ] CI validation for translation PRs (frontmatter, structure, code blocks)

### Software i18n
- [ ] Zero hardcoded user-facing strings
- [ ] Plural forms handled via i18n library (not string concat)
- [ ] Dates/numbers formatted with `Intl.*` or locale-aware API
- [ ] RTL layout tested if targeting Arabic/Hebrew/Persian users
- [ ] Locale detection from environment (CLI) or browser (web)
- [ ] Fallback locale defined (always `en`)
- [ ] Translation tooling configured (Crowdin / Weblate / manual)
- [ ] Source locale file is source of truth — never edit translations directly
